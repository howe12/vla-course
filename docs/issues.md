## 1. `<pre><code>` 真实换行符破坏 JS

### 现象

页面 JS 报 `Uncaught SyntaxError`，页面卡在"载入中"，所有章节无法渲染。

### 根因

`index.html` 中所有章节内容以 JS 模板字符串形式存放。当 `<pre><code>` 块内包含**真实换行符**（而非 `\n` 转义）时，JS 解析器将换行视为语句结束，导致语法错误。

```javascript
// ❌ 错误：真实换行符
'<pre><code>#!/usr/bin/env python3
import torch
print("hello")</code></pre>'

// ✅ 正确：\n 转义
'<pre><code>#!/usr/bin/env python3\\nimport torch\\nprint("hello")</code></pre>'
```

### 时间线

| 提交 | 日期 | 说明 |
|------|------|------|
| `a107bde` | ~6/28 | 修复 §3 L40 gpu_test.py 代码块的换行符 |
| — | ~6/29 | 从 `aa03b73` 恢复 Ch1 时再次引入换行符问题 |
| `34382d3` 前 | ~6/29 | Ch1 第0步 `</code>` 后双引号 + 转义错误 |

### 教训

- **任何涉及 HTML 编辑的脚本**必须检查 `<pre><code>` 内容是否已 `\n` 转义
- 从 Git 历史恢复内容时，原始版本可能就含真实换行（当时没触发是因为页面还没上线）
- **建议**：写一个 lint 脚本自动检测 `index.html` 中 `<pre><code>` 块的真实换行

## 2. 字符串拼接缺 `+` 导致整脚本报废

### 现象

Ch1 内容反复丢失——恢复后过段时间又只剩 1.2 之前的内容。

### 根因

`r_ch1()` 函数使用单引号字符串 `+` 拼接方式构建 HTML。**3 处缺 `+`** 导致相邻字符串字面量无运算符，JS 解析器报 `Unexpected string`。

更致命的是：整个课程的所有章节渲染函数（`r_ch0` 到 `r_ch8`）都在**同一个 `<script>` 块**中。一处语法错误 → 整块报废 → 所有章节渲染函数 `undefined` → 页面降级到 Jekyll 预渲染的静态兜底 HTML。

### 3 处缺 `+` 位置

| 行号 | 内容 | 影响 |
|------|------|------|
| ~357 | `'</section>'` → 应 `'</section>'+` | 1.3~1.13 全部不渲染 |
| ~477 | `'<h3>1.12...</h3>'` → 应 `...'+` | 1.12 内容不渲染 |
| ~482 | `'</section>'` → 应 `'</section>'+` | 1.13 + 小结 + 思考题不渲染 |

### 时间线

| 提交 | 说明 |
|------|------|
| `34382d3` | 修复第 1 处（1.2 card 后的 `</section>'`）|
| `4fb0dd1` | 修复第 2、3 处（1.12 标题和结尾）|

### 为什么反复出现？

1. 每次从 Git 历史恢复 Ch1 时，源内容（`aa03b73`）就含这 3 个 bug
2. 修复 JS 语法时只关注了 `<pre><code>` 换行，没检查字符串拼接
3. 页面在 Jekyll 预渲染时**部分成功**（到 1.2），给人一种"修好了"的假象

### 教训

- **绝对不能依赖"页面能显示一部分"来判断 JS 是否 OK**
- 必须用 `node -e "new Function(scriptContent)"` 验证整个脚本块
- 提交前增加 **JS 语法检查 CI 步骤**

## 7. 浏览器缓存掩盖修复

### 现象

修复推送到 GitHub Pages 后，浏览器仍显示旧版本。多次误判"修复无效"。

### 根因

GitHub Pages 默认缓存策略（`cache-control: max-age=600`）+ 浏览器自身缓存。

### 规避方法

- 用 `curl` 直接验证部署版本：`curl -s https://howe12.github.io/vla-course/index.html | grep <关键字>`
- 浏览器验证时加随机参数或用隐私模式
- 检查 GitHub Actions 部署状态确认构建完成
